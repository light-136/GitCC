# SmartMES 项目目录逐层解读

> 目的：帮助阅读者从“目录结构”的角度理解 `SmartMES` 项目，快速知道每一层、每个工程、每个主要文件夹分别负责什么。

---

## 1. 项目根目录总览

项目根目录如下：

```text
SmartMES/
├── .vscode/
├── docs/
├── SmartMES.Core/
├── SmartMES.Modules/
├── SmartMES.Native/
├── SmartMES.Services/
├── SmartMES.Tests/
├── SmartMES.UI/
├── .gitignore
├── README.md
└── SmartMES.sln
```

从这个结构就能看出，这是一个**标准的多项目解决方案**，不是单一工程。

根目录中的内容可以分成三类：

1. **工程与源码**
   - `SmartMES.Core`
   - `SmartMES.Services`
   - `SmartMES.Modules`
   - `SmartMES.UI`
   - `SmartMES.Tests`
   - `SmartMES.Native`

2. **文档与说明**
   - `README.md`
   - `docs/`

3. **开发环境配置**
   - `.vscode/`
   - `SmartMES.sln`
   - `.gitignore`

---

## 2. 根目录文件说明

### 2.1 `README.md`

这是项目对外的总入口说明。

主要包含：

- 项目名称和版本
- 快速启动方式
- 内置账号
- 系统主要功能
- 项目结构概览
- 技术栈
- 文档索引

如果第一次接触项目，应该先看这里。

---

### 2.2 `SmartMES.sln`

这是 Visual Studio 解决方案文件。

它把多个子工程组织在一起，包括：

- `SmartMES.Core`
- `SmartMES.Services`
- `SmartMES.Modules`
- `SmartMES.UI`
- `SmartMES.Tests`

它的作用是：

- 统一构建整个项目
- 管理项目间引用关系
- 在 IDE 中作为整体工程打开

你可以把它理解成“整个系统的总装配入口”。

---

### 2.3 `.gitignore`

用于忽略构建产物、缓存文件、IDE 配置等不需要纳入版本管理的内容。

---

### 2.4 `.vscode/`

包含：

- `launch.json`
- `tasks.json`

说明项目为 VS Code / Cursor 这类编辑器配置了：

- 启动方式
- 任务执行方式

它不属于业务代码，但对开发调试很有帮助。

## 4. `SmartMES.Core/` 核心基础层解读

这是整个项目的**底层基础能力工程**。

它不直接做复杂 UI，也不主要承载业务页面，而是放置整个系统都会用到的核心抽象与公共能力。

目录结构：

```text
SmartMES.Core/
├── Infrastructure/
├── Interfaces/
├── IO/
├── Models/
├── Plugin/
├── Recipe/
├── Safety/
├── Scheduler/
├── StateMachine/
├── Traceability/
└── SmartMES.Core.csproj
```

---

### 4.1 `Infrastructure/`

包含：

- `RelayCommand.cs`
- `ViewModelBase.cs`

这是典型 MVVM 基础设施。

作用：

- `ViewModelBase`：提供属性变更通知
- `RelayCommand`：为按钮命令、菜单命令提供绑定能力

这部分是 UI 层 ViewModel 的基础依赖。

---

### 4.2 `Interfaces/`

包含：

- `IAlarmService.cs`
- `ICommunicationService.cs`
- `IDevice.cs`
- `IEventBus.cs`
- `ILoggingService.cs`
- `ISettingsService.cs`

这是系统的**接口定义层**。

作用：

- 给服务层和 UI 层之间提供抽象契约
- 降低模块耦合
- 方便替换具体实现
- 方便测试

可以把这里理解为“系统能力的统一协议定义”。

---

### 4.3 `IO/`

包含：

- `IoDevice.cs`

用于表达 IO 设备或 IO 点位相关的基础抽象。

在工业场景中，IO 往往代表：

- 输入信号
- 输出信号
- 传感器状态
- 执行器状态

这一层属于工业控制基础模型的一部分。

---

### 4.4 `Models/`

包含：

- `AlarmRecord.cs`
- `DeviceModel.cs`
- `LogEntry.cs`
- `SystemSettings.cs`
- `UserModel.cs`

这里放的是系统通用数据模型。

例如：

- 报警记录
- 设备信息
- 日志记录
- 系统配置
- 用户和权限

这部分属于“跨层共享模型”。

---

### 4.5 `Plugin/`

包含：

- `PluginLoader.cs`

这是项目的插件化基础设施。

作用：

- 扫描目录中的插件 DLL
- 加载实现指定接口的程序集
- 初始化插件
- 卸载插件
- 广播调用插件逻辑

这说明项目不是完全封闭结构，而是预留了扩展能力。

---

### 4.6 `Recipe/`

包含：

- `RecipeService.cs`

用于管理工艺配方、产品参数集等内容。

在制造业中，配方系统通常用于：

- 不同产品切换时加载不同参数
- 保存标准工艺参数
- 统一管理工艺版本

这是典型工业系统能力之一。

---

### 4.7 `Safety/`

包含：

- `SafetyService.cs`

表示安全互锁、设备保护、条件判断等相关逻辑。

这部分通常用于：

- 急停保护
- 条件联锁
- 危险动作阻止
- 安全状态判定

---

### 4.8 `Scheduler/`

包含：

- `TaskSchedulerService.cs`

这是统一任务调度中心。

从代码判断，它支持：

- 一次性任务
- 周期任务
- 延迟任务
- 优先级任务
- 任务执行状态跟踪

这一层很适合承载工业后台任务，例如：

- 设备轮询
- 定时采集
- 定时上报
- 自动流程触发

---

### 4.9 `StateMachine/`

包含：

- `StateMachineEngine.cs`

这是工业控制领域非常核心的部分。

它提供：

- 状态定义
- 状态迁移
- 触发器
- 状态切换事件
- 标准工业设备状态流转模板

这意味着系统底层逻辑不是随意写 if/else，而是在朝标准状态管理方向设计。

---

### 4.10 `Traceability/`

包含：

- `TraceService.cs`

这是追溯系统。

主要用于：

- 按 SN 跟踪产品
- 记录工序执行过程
- 保存参数与结果
- 判断最终 OK/NG
- 提供查询能力

这是 MES / 制造执行领域的重要基础模块。

---

### 4.11 `SmartMES.Core.csproj`

核心工程项目文件。

主要说明：

- 目标框架：`net9.0-windows`
- 开启可空引用与隐式 using
- 依赖 `System.Text.Json`

---

## 5. `SmartMES.Services/` 服务层解读

这是系统的**公共服务实现层**。

目录结构：

```text
SmartMES.Services/
├── Alarm/
├── Automation/
├── Communication/
├── Device/
├── EventBus/
├── Logging/
├── SettingsService.cs
└── SmartMES.Services.csproj
```

这一层的特点是：

- 向上服务 UI
- 向下依赖 Core 抽象
- 把“系统级能力”组织成可复用服务

---

### 5.1 `Alarm/`

包含：

- `AlarmService.cs`

负责报警触发、管理和通知等逻辑。

---

### 5.2 `Automation/`

包含：

- `AutomationEngine.cs`
- `AutomationStep.cs`

这是自动化流程引擎。

作用：

- 定义流程步骤
- 顺序执行步骤
- 记录流程状态
- 流程完成通知
- 失败中断处理

这是设备联动、工序自动执行演示的重要基础。

---

### 5.3 `Communication/`

包含：

- `SerialCommunicationService.cs`
- `TcpCommunicationService.cs`

这是基础通信服务层。

用于承载串口、TCP 等通用通信逻辑。

---

### 5.4 `Device/`

包含：

- `PlcDevice.cs`
- `SensorDevice.cs`

说明服务层包含设备级抽象实现。

可理解为：

- PLC 设备对象
- 传感器设备对象
- 设备连接与行为模拟或封装

---

### 5.5 `EventBus/`

包含：

- `EventBusService.cs`

用于实现事件发布/订阅。

作用：

- 降低模块间直接耦合
- 让 UI、服务、模块通过事件通信
- 支持登录、状态变化、消息通知等场景

---

### 5.6 `Logging/`

包含：

- `LoggingService.cs`

用于统一日志记录。

工业系统中日志很关键，因为它直接关联：

- 排障
- 审计
- 操作回溯
- 设备行为跟踪

---

### 5.7 `SettingsService.cs`

负责系统参数读取、保存和管理。

---

### 5.8 `SmartMES.Services.csproj`

服务层工程文件。

说明：

- 依赖 `SmartMES.Core`
- 使用 `System.Text.Json`
- 目标框架是 `net9.0-windows`

## 6. `SmartMES.Modules/` 业务模块层解读

这是整个项目中最能体现“业务能力”的一层。

目录结构：

```text
SmartMES.Modules/
├── Database/
├── FileProcess/
├── MesComm/
├── MotionControl/
├── NativeInterop/
```

这一层负责把具体功能做出来。

---

### 6.1 `Database/`

包含：

- `DbContext.cs`
- `Repositories.cs`

负责数据库相关实现。

可见功能：

- 定义实体
- 配置 EF Core 上下文
- 支持 SQLite / MySQL / SQL Server
- 提供数据库访问基础

---

### 6.2 `FileProcess/`

包含：

- `ExcelFileService.cs`
- `FileServices.cs`

负责文件读写和导出。

大概率承担：

- Excel 导出
- CSV/JSON 文件操作
- 批量文件处理

---

### 6.3 `MesComm/`

包含：

- `HttpMesClient.cs`
- `IMesClient.cs`
- `MqttMesClient.cs`
- `OpcUaMesClient.cs`

这是项目的 MES / 工业通信模块。

意义在于：

- 对接外部系统
- 拉取工单
- 上报生产结果
- 接收系统推送
- 为后续真实通信扩展预留统一接口

---

### 6.4 `MotionControl/`

包含：

- `AxisController.cs`
- `MultiAxisController.cs`
- `TenAxisController.cs`

这是运动控制逻辑核心。

按层级理解：

- `AxisController`：单轴

### 6.5 `NativeInterop/`

包含：

- `NativeMath.cs`

作用：

- 封装 C++ 原生 DLL 导出函数
- 为 UI 或模块层提供易用 API
- 判断 native 库是否可用

这是托管层与原生层之间的桥梁。

---

### 6.6 `Vision/`

包含：

- `VisionEngine.cs`

负责视觉算法示例逻辑。

从代码看，包括：

- 模拟图像生成
- 灰度化
- Sobel 检测
- 阈值缺陷检测
- 标注绘制

---

### 6.7 `VisionMotion/`

包含：

- `MultiStationScheduler.cs`

这个目录体现了“视觉 + 运动 + 调度”的综合场景。

说明项目不仅有单一模块，还在尝试模拟多工位协同控制。

---

### 6.8 `SmartMES.Modules.csproj`

模块工程文件。

可见特征：

- 依赖 `SmartMES.Core`
- 开启 `AllowUnsafeBlocks`
- 使用 `UseWPF`
- 引入 `EF Core`、`Dapper`、`EPPlus` 等包

这一点也说明：模块层不仅处理业务逻辑，还承担了一部分与 WPF 数据结构兼容的职责，例如视觉图像处理里会直接使用 WPF 图像对象。

## 9. `SmartMES.Native/` 原生工程解读

目录包含：

- `build.ps1`
- `CMakeLists.txt`
- `SmartMES.Native.cpp`

这是一个独立的 C++ 原生动态库工程。

### 9.1 `SmartMES.Native.cpp`

实现原生算法与导出函数。

### 9.2 `CMakeLists.txt`

用于 C++ 工程构建。

从配置看：

- 编译共享库
- 使用 `C++17`
- 输出到 `SmartMES.UI` 的 bin 目录

说明其目的就是让 UI 启动时可直接加载这个 DLL。

### 9.3 `build.ps1`

## 10. 目录层级关系怎么理解

如果用一句话概括整个目录关系：

```text
UI 负责展示
Services 负责公共系统服务
Modules 负责具体业务能力
Core 提供基础抽象与底层能力
Native 提供 C++ 扩展
Tests 负责验证
Docs 负责说明
```

也可以理解成下面这条链路：

```text
用户操作
  -> UI 页面 / ViewModel
  -> Services 公共服务协调
  -> Modules 具体业务实现
  -> Core 基础能力支撑
  -> Native 提供高性能或原生扩展
```

## 12. 最终结论

从目录结构上看，`SmartMES` 不是一个简单演示页面集合，而是一个组织较完整的工业软件多层项目。

它的目录设计体现了几个特点：

1. **分层清楚**：基础层、服务层、业务层、表现层边界较明确。
2. **模块化明显**：数据库、MES、运动、视觉、原生扩展都独立成模块。
3. **工程化较好**：有文档、有测试、有原生工程、有 UI 资源组织。
4. **适合扩展**：插件化、接口抽象、事件总线、多数据库支持都为扩展留了空间。

如果从“目录设计质量”来看，这个项目已经具备一个较成熟原型系统或样板工程的结构基础。
