# 工业上位机通信系统专题精讲

> 文档定位：专题教学文档  
> 目标：系统讲清工业上位机中的通信系统为什么重要、通信架构如何设计、常见协议怎么分层理解、真实项目中会遇到什么问题，以及 `SmartMES` 在其中的映射。  
> 适用对象：希望真正理解工业通信，而不是只会写几个 socket 或串口示例的人。

---

# 1. 为什么通信系统是工业上位机的第一核心？

因为上位机本身并不直接感知世界。它绝大多数信息和控制能力都来自“通信”。

没有通信，上位机就无法：

- 获取设备状态
- 读取传感器数据
- 写入控制命令
- 获取工单
- 上传结果
- 同步系统状态

所以你可以这样理解：

> **通信系统，是上位机连接现实设备与外部系统的神经网络。**

如果通信做不好，上位机再漂亮、页面再多，也只是一个“好看的空壳”。

---

# 2. 工业通信到底在“连”什么？

工业上位机的通信对象通常分成两大类：

## 2.1 向下通信：连接现场设备

常见对象包括：

- PLC
- 传感器
- IO 模块
- 运动控制卡
- 工业相机
- 仪器仪表
- 条码枪 / 扫码器
- 光源控制器
- 伺服驱动器

这些通信的目的通常是：
- 读状态
- 写命令
- 取测量值
- 取报警
- 取参数

## 2.2 向上通信：连接管理系统

常见对象包括：

- MES
- SCADA
- 数据平台
- Web API
- 云平台
- 报表系统

这些通信的目的通常是：
- 获取工单
- 上报结果
- 同步产量
- 回传状态
- 上传追溯记录

# 3. 工业通信和普通互联网通信最大的不同是什么？

很多初学者会把工业通信理解成“换个协议而已”，其实差别很大。

## 3.1 工业通信更关心状态和稳定性

互联网开发经常是请求-响应模式。  
工业通信更常见的是：

- 长连接
- 周期采集
- 心跳
- 断线重连
- 连续状态同步
- 实时性要求

## 3.2 工业通信更强依赖物理对象

你不是在和一个抽象服务说话，而是在和真实设备打交道。  
这意味着：

- 对方可能掉电
- 对方可能线缆松动
- 对方可能状态机卡死
- 对方可能响应慢
- 对方可能不是标准实现

## 3.3 工业通信更强调容错与可诊断

因为现场故障往往不是“接口报 500”那么简单，而是：
- 偶发丢包
- CRC 错误
- 半连接
- 设备假在线
- 逻辑在线但数据异常

所以工业通信本质上比普通 CRUD 通信要复杂得多。

---

# 4. 一个完整的通信系统，不能只理解成“收发数据”

这是最重要的认知之一。

成熟的工业通信系统，通常至少有 5 层。

## 4.1 物理/链路层

这是通信的底层通道，比如：
- 串口线
- 网线
- 工业交换机
- RS232 / RS485
- 以太网

这层更偏硬件，但软件要感知它的存在，因为很多问题就出在这里。

## 4.2 协议层

这是“说话方式”，比如：
- Modbus RTU
- Modbus TCP
- OPC UA
- MQTT
- HTTP REST
- 厂商私有协议

## 4.3 连接管理层

负责：
- 建立连接
- 断开连接
- 心跳维护
- 超时控制
- 自动重连
- 连接状态通知

## 4.4 消息封装层

负责：
- 报文编码
- 报文解码
- 序列化/反序列化
- 错误码解释
- 数据结构转换

## 4.5 业务语义层

这是最重要的一层。

它把“底层报文”翻译成“业务能力”。

例如：
- 不是“读地址 40001”，而是“读取工件温度”
- 不是“写寄存器 12”，而是“启动设备”

所以成熟系统里，通信不是一个工具类，而是一整套系统。

# 5. 工业通信最常见的 6 种方式

下面我按学习优先级给你解释。

# 6. 串口通信

## 6.1 串口是什么

串口是一种非常常见、非常基础的设备通信方式，尤其在：
- 仪器仪表
- 扫码器
- 一些老设备
- 一些控制器
中非常常见。

## 6.2 串口要关注什么

- COM 口号
- 波特率
- 数据位
- 停止位
- 校验位
- 超时
- 收发缓存

## 6.3 串口通信常见问题

- 端口占用
- 波特率不一致
- 数据乱码
- 收到半包
- 缓冲区没清理
- 线接错

## 6.4 串口适合什么场景

适合简单设备、点对点连接、小体量数据交互。

## 6.5 学习重点

串口虽然看起来“老”，但它是学习工业通信非常好的起点，因为你能清晰理解：
- 物理连接
- 报文
- 超时
- 状态

## 6.6 SmartMES 映射

- `SerialCommunicationService`

---

# 7. TCP 通信

## 7.1 TCP 是什么

TCP 是基于网络的稳定连接方式，工业上大量设备都通过 TCP/IP 通信。

## 7.2 工业 TCP 的典型用途

- 与 PLC 通信
- 与控制器通信
- 与上层系统通信
- 与嵌入式设备通信

## 7.3 TCP 通信系统必须考虑什么

- 主动连接还是被动监听
- 长连接还是短连接
- 超时
- 心跳
- 半包/粘包
- 断线重连
- 并发访问

## 7.4 为什么 TCP 不是“连上就完事”

因为真实现场里最常见的问题不是“完全断开”，而是：
- 连接还在，但对方不响应
- 网络波动导致半死不活
- 一会儿能收一会儿不能收
- 对方重启后连接状态失真

所以 TCP 在工业项目里一定要配连接管理逻辑。

## 7.5 SmartMES 映射

- `TcpCommunicationService`

---

# 8. Modbus：工业现场最经典的协议之一

## 8.1 Modbus 为什么重要

如果你学工业通信，Modbus 几乎绕不开。它是很多设备之间最常用的基础协议之一。

## 8.2 Modbus 有哪些常见形式

- Modbus RTU：走串口
- Modbus TCP：走网络

## 8.3 Modbus 的本质

它本质上是：

> 基于地址空间读写寄存器和线圈。

例如：
- 读取线圈状态
- 读取保持寄存器
- 写单个寄存器
- 写多个寄存器

## 8.4 为什么说 Modbus 很适合学习工业通信

因为它很典型地体现了工业通信的特点：
- 面向地址
- 面向设备寄存器
- 协议简单但场景广泛
- 需要你自己做业务语义封装

## 8.5 学习重点

你要意识到：
- 协议本身只提供“读写地址”
- 真正的业务意义，需要上位机自己解释

比如：
- 地址 40001 代表速度
- 地址 40002 代表温度
- 地址 40003 代表报警码

所以上位机里通常还需要“地址映射表”。

---

# 9. OPC UA：现代工业语义通信的重要代表

## 9.1 OPC UA 是什么

OPC UA 是现代工业非常重要的标准化通信协议。它不只是传输数据，更强调：
- 语义
- 结构
- 安全
- 统一建模

## 9.2 它和 Modbus 最大的不同

Modbus 更像“读地址”。  
OPC UA 更像“读对象和节点”。

也就是说，OPC UA 更适合表达复杂工业对象结构。

## 9.3 OPC UA 的优势

- 标准化程度高
- 面向对象建模
- 安全性更好
- 适合多系统集成
- 适合大型工业场景

## 9.4 OPC UA 学习重点

你要理解：

> OPC UA 的价值，不仅在通信，更在统一工业信息模型。

## 9.5 SmartMES 映射

- `OpcUaMesClient`

虽然当前更偏结构预留，但方向是对的。

---

# 10. MQTT：面向消息和轻量分发的工业通信方式

## 10.1 MQTT 是什么

MQTT 是一种轻量级发布/订阅消息协议，在工业物联网里很常见。

## 10.2 为什么工业里会用 MQTT

因为它适合：
- 轻量消息传输
- 事件分发
- 多节点订阅
- 云边协同

## 10.3 MQTT 的核心思维

不是请求-响应，而是：

- 发布到主题
- 多方订阅主题
- 消息按主题流动

## 10.4 MQTT 适合做什么

- 状态广播
- 事件通知
- 轻量遥测
- 边缘到云端的数据同步

## 10.5 SmartMES 映射

- `MqttMesClient`

---

# 11. HTTP REST：最常见的系统间业务接口方式

## 11.1 HTTP REST 在工业里通常干什么

- 获取工单
- 上传结果
- 获取配置
- 调用管理接口

## 11.2 它的优势

- 生态成熟
- 实现简单
- 容易和企业系统集成
- 文档清晰

## 11.3 它的局限

- 不适合强实时控制
- 更适合业务数据交互
- 通常适合上位机与 MES/平台间通信，而非底层实时设备控制

## 11.4 SmartMES 映射

- `HttpMesClient`

这个模块很好地说明：

> HTTP 在上位机里更像“业务协同接口”，而不是“实时控制通道”。

# 12. 为什么“协议”和“系统通信设计”不是一回事？

这点非常关键。

很多初学者学完几个协议，就觉得会做通信系统了。其实还差很远。

协议只回答：
- 数据怎么传

而一个完整通信系统还要回答：
- 谁负责连
- 连不上怎么办
- 断了怎么办
- 超时怎么办
- 数据错了怎么办
- 状态怎么显示
- 日志怎么打
- 如何让业务层易用

所以：

> 协议是基础，通信系统设计才是工程能力。

# 13. 工业通信系统设计的标准结构，我建议你这样记

我推荐你以后设计通信系统时，按这 6 层思考：

## 第 1 层：物理连接层
串口 / 网口 / 工业网络基础。

## 第 2 层：协议层
Modbus / OPC UA / MQTT / HTTP / 私有协议。

## 第 3 层：连接管理层
连接、断线、心跳、重连、超时。

## 第 4 层：报文处理层
打包、解包、序列化、校验。

## 第 5 层：设备/业务封装层
把协议细节转成：
- 读取温度
- 获取工单
- 上报结果
- 写入启动命令

## 第 6 层：可视化与诊断层
状态灯、通信日志、错误提示、报文历史、最近错误摘要。

只要你能把这 6 层理顺，通信系统设计基本就不会跑偏。

# 14. 通信系统最容易出问题的 10 个地方

## 14.1 断线没有重连
## 14.2 假在线
## 14.3 收到半包或粘包
## 14.4 超时处理不清晰
## 14.5 通信异常没有进入日志
## 14.6 通信线程直接改 UI
## 14.7 多线程同时操作同一连接
## 14.8 协议细节泄漏到 UI 层
## 14.9 没有心跳机制
## 14.10 没有明确的连接状态模型

这 10 个问题你以后做通信时一定会遇到，提前知道能少走很多弯路。

# 15. 通信状态为什么一定要“建模”？

很多人通信只用一个 bool：
- `IsConnected = true/false`

这其实不够。

一个更合理的通信状态模型，至少可能包括：
- Disconnected
- Connecting
- Connected
- Reconnecting
- Faulted
- Timeout

为什么？
因为工业现场里“在线”不是非黑即白。很多时候需要让用户知道系统到底处于哪种通信状态。

---

# 16. 工业通信日志为什么要单独设计？

通信日志不同于普通日志，它通常必须回答：

- 什么时候连上的
- 什么时候断的
- 连了哪个地址
- 发了什么命令
- 收到了什么响应
- 超时发生在哪次调用
- 错误码是什么

如果没有这些信息，现场排障会非常困难。

所以好的上位机通常都有：
- 通信日志分类
- 通信状态页
- 报文调试页
- 错误摘要

# 17. 通信系统和其他模块是怎么协同的？

通信不是独立模块，它会和很多模块强关联。

## 与设备抽象模块协同
通信负责把真实数据带回来，设备模块负责把它转成对象状态。

## 与 UI 协同
通信状态要显示给用户。

## 与报警协同
断线、超时、非法响应可能触发报警。

## 与日志协同
所有关键通信事件都应记录。

## 与流程协同
流程执行往往要依赖通信成功。

## 与追溯协同
某些通信结果会变成追溯记录的一部分。

这说明：

> 通信是横向能力，不是孤立工具模块。

# 18. 从 SmartMES 看，通信部分最值得学什么？

我认为有三个重点：

## 18.1 不同通信对象分成不同客户端实现
比如：
- HTTP
- MQTT
- OPC UA

这体现了“协议适配器”思路。

## 18.2 通信与业务不是完全混在一起
`HttpMesClient` 已经开始体现“业务客户端”的思路，而不是把所有请求散在页面里。

## 18.3 通信是被纳入 UI 功能体系的
项目里有通信监控页面，这说明通信不是“后台黑盒”，而是用户可见、可诊断的系统功能。

# 19. 如果你以后自己设计通信系统，我建议你这样做

## 第一步：先定义通信对象
- PLC？
- 相机？
- MES？
- 控制器？

## 第二步：再定义协议
- 串口？
- TCP？
- Modbus？
- OPC UA？
- HTTP？

## 第三步：定义连接状态模型
不要只用一个 `bool`。

## 第四步：定义错误模型
超时、断线、非法响应、认证失败、解析失败要能区分。

## 第五步：定义通信日志
通信必须可追踪。

## 第六步：把协议细节封装起来
UI 不应该知道寄存器地址和报文字节数组。

## 第七步：设计重连和恢复机制
通信系统必须考虑现场不稳定性。

---

# 20. 一句话总结工业通信

> 工业上位机中的通信系统，不是简单的“发消息和收消息”，而是一整套连接管理、协议处理、状态建模、错误恢复、业务封装与可视化诊断的工程体系；学会通信，才算真正推开了工业上位机的大门。
