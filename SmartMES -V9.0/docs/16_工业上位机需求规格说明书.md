## 1. 引言

### 1.1 编写目的

本文档用于定义 `SmartMES` 工业上位机系统的完整需求范围，回答以下问题：

- 系统要解决什么问题；
- 系统面向哪些用户；
- 系统需要提供哪些功能；
- 系统应满足哪些性能、安全、稳定性和可维护性要求；
- 后续如何进行设计、开发、测试和验收。

### 1.2 项目背景

在智能制造场景中，现场上位机承担着连接设备、连接人员、连接工艺和连接上层系统的重要作用。传统现场软件往往存在以下问题：

- 功能分散，缺乏统一操作入口；
- 设备、数据、质量、工艺等信息分散在多个工具中；
- 无法形成完整的生产追溯链；
- 缺少可扩展的软件架构，后期维护困难；
- 缺少统一的报警、日志和权限机制。

因此，需要建设一套统一的工业上位机系统，实现制造现场的“可监控、可操作、可记录、可追溯、可扩展”。

### 1.3 系统定义

`SmartMES` 是一套面向制造现场的 Windows 桌面工业上位机系统，基于 `WPF + .NET 9 + MVVM` 架构开发，用于实现：

- 设备监控
- 通信管理
- 报警与日志
- 自动化流程
- MES 数据交互
- 数据库管理
- 文件导入导出
- 运动控制
- 视觉检测
- 安全互锁
- 过程追溯
- 配方管理
- 权限管理

## 2. 总体描述

### 2.1 产品定位

本系统定位为：

1. 智能制造现场的工业上位机；
2. 制造执行过程中的桌面操作终端；
3. 工业项目原型和二次开发基础平台；
4. 适合演示、教学、汇报和实施扩展的综合系统。

### 2.2 产品目标

系统建设目标如下：

1. 建立统一的人机交互入口；
2. 将设备、工艺、质量、数据、权限纳入统一系统；
3. 提供完整的基础工业能力内核；
4. 支持从模拟演示平滑演进到真实接入；
5. 保障系统具备可维护性、可测试性和可扩展性。

### 2.3 用户角色

系统至少包含以下角色：

#### 2.3.1 管理员
- 管理所有页面；
- 管理用户与权限；
- 修改系统配置；
- 查看全部日志与报警；
- 具备全部操作权限。

#### 2.3.2 操作员
- 使用日常生产相关页面；
- 执行流程、控制设备、查看状态；
- 不可修改关键系统配置或用户权限。

#### 2.3.3 观察者
- 查看关键状态和结果；
- 以只读方式访问部分页面；
- 不执行高风险操作。

### 2.4 系统运行环境

- 操作系统：Windows 10 / 11
- 框架：.NET 9
- UI：WPF
- 数据库：SQLite / MySQL / SQL Server
- 部署方式：桌面应用本地运行

### 2.5 系统边界

#### 上行系统
- MES
- 数据平台
- 报表平台
- 可选云端平台

#### 下行对象
- PLC
- 传感器
- IO 模块
- 执行器
- 工业相机
- 运动控制模块
- 串口 / TCP / MQTT / OPC UA 设备

### 2.6 假设与依赖

1. 现场具备 Windows 桌面运行环境；
2. 安装了 .NET 运行时；
3. 若使用 MySQL 或 SQL Server，则现场已具备对应数据库服务；
4. 若接入真实 MES，则现场已提供接口文档；
5. 若接入真实运动/视觉设备，则现场已提供驱动或 SDK。

## 3. 业务需求

### 3.1 统一操作需求

现场人员需要一个统一入口查看设备状态、执行流程、查看报警、管理数据，而不是在多个独立工具之间切换。

### 3.2 统一状态感知需求

系统需要让用户在短时间内感知：

- 设备是否在线；
- 系统是否报警；
- 当前生产是否正常；
- 关键模块是否可用；
- 当前登录用户具备哪些权限。

### 3.3 过程可追溯需求

每个产品或工单的加工过程，需要能够记录：

- 开始时间；
- 工序步骤；
- 工位与设备；
- 参数数据；
- 检测结果；
- 操作人；
- 最终 OK/NG。

### 3.4 风险可控制需求

高风险操作必须有以下保证：

- 权限校验；
- 安全条件校验；
- 急停优先；
- 报警联动；
- 日志留痕。

### 3.5 模块可扩展需求

系统不能写死为单一项目，而应支持：

- 新增设备；
- 新增页面；
- 新增协议；
- 新增算法模块；
- 新增客户定制能力。

## 5. 非功能需求

## 5.1 性能需求

### NFR-001 页面响应
一般页面切换应在用户可接受范围内完成，通常不应出现明显卡顿。

### NFR-002 数据处理响应
普通统计和数据加载在演示规模数据量下应能快速完成。

### NFR-003 视觉处理响应
视觉模块在当前示例分辨率下应具备可演示的实时性。

---

## 5.2 稳定性需求

### NFR-004 异常防护
系统应对 UI 线程异常、后台任务异常和进程级异常提供兜底处理。

### NFR-005 长时间运行
系统应支持长时间运行，不应频繁崩溃或严重内存泄漏。

### NFR-006 急停优先
急停相关逻辑必须高优先级处理。

---

## 5.3 安全需求

### NFR-007 权限控制
不同角色只应访问被授权的页面与功能。

### NFR-008 高风险操作保护
高风险操作必须进行权限与安全条件校验。

### NFR-009 日志审计基础
关键操作必须具备日志留痕。

---

## 5.4 可维护性需求

### NFR-010 分层架构
系统必须保持分层边界清晰。

### NFR-011 模块可替换
模拟实现和真实实现应能沿统一接口逐步替换。

### NFR-012 可读性
代码应具备统一命名、注释和编码规范。

---

## 5.5 可测试性需求

### NFR-013 核心逻辑可测试
状态机、调度、配方、安全、追溯、运动控制等核心逻辑应可独立测试。

### NFR-014 权限逻辑可验证
导航权限与页面回退逻辑应具备自动化测试覆盖。

---

## 5.6 可扩展性需求

### NFR-015 支持新增模块
系统应能新增页面、协议、算法和设备模块。

### NFR-016 支持插件化方向
系统应为插件扩展预留技术边界。

## 8. 约束条件

### 8.1 技术约束
- 当前采用 WPF 桌面模式；
- 当前运行环境为 Windows；
- 当前主要技术栈为 C# / .NET 9；
- 当前项目架构为多项目分层架构。

### 8.2 项目约束
- 不应破坏现有模块化结构；
- 不应将复杂业务逻辑直接写入 View 层；
- 高风险控制逻辑必须纳入安全互锁范围；
- 新增模块需遵守统一权限、日志和配置规则。

## 9. 验收标准

### 9.1 功能验收
1. 系统能正常构建与运行；
2. 主要页面均可访问；
3. 角色权限控制生效；
4. 报警、日志、数据库、运动、视觉等核心模块可演示；
5. 状态机、调度、安全、追溯等基础能力可正常工作。

### 9.2 质量验收
1. 关键异常不导致系统直接崩溃；
