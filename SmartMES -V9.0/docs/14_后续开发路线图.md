# SmartMES 后续开发路线图

> 目的：基于当前 `SmartMES` 项目现状，给出一份清晰、可执行的后续开发路线图，帮助项目从“高质量演示平台 / 架构样板”逐步演进为“更真实、可部署、可扩展的工业上位机系统”。

---

## 1. 当前项目状态判断

结合现有代码、文档和目录结构，`SmartMES` 当前已经具备以下基础：

### 已有优势

- 具备完整的多项目分层架构
- 已有 WPF 桌面 UI 与 MVVM 基础设施
- 已实现报警、日志、自动化流程、数据库、MES 通信、运动控制、视觉检测等模块
- 已有状态机、调度器、追溯、配方、安全互锁等工业底层能力
- 已具备部分测试和文档沉淀
- 已预留插件扩展能力与原生 C++ 扩展能力

### 当前短板

- 多数模块仍偏“演示 / 模拟”实现
- 与真实硬件、真实工业协议、真实 MES 的接入深度有限
- 部署、配置、诊断、运维能力还不完整
- 缺少统一的依赖注入、配置中心、模块生命周期治理
- 缺少面向长期工业现场运行的稳定性设计

### 因此当前最准确的阶段定位是：

> **V1：架构样板 + 工业综合演示平台**

后续开发的核心目标，就是把它逐步推进到：

> **V2：真实业务可接入的工业原型系统**  
> **V3：可部署、可维护、可扩展的工程化上位机平台**

---

## 2. 总体演进目标

建议把项目演进分为 4 个阶段：

1. **阶段 A：夯实基础架构**
2. **阶段 B：把演示模块升级为真实可接入模块**
3. **阶段 C：增强工程化与现场可运维能力**
4. **阶段 D：平台化、产品化、可复制化**

这样做的好处是：

- 不会一开始就把目标定得过大
- 能保留现有演示成果
- 可以边做边验证
- 有利于逐步接入真实项目需求

---

# 阶段 A：夯实基础架构

## 3. A 阶段目标

目标不是增加很多新功能，而是先把“系统底座”打牢。

### 阶段目标

- 统一项目组织方式
- 提升代码可维护性
- 清理模拟逻辑与真实逻辑的边界
- 强化模块间解耦
- 为后续真实接入做准备

---

## 4. A1：统一依赖注入与模块注册

### 当前情况

从现有结构看，项目已经有接口与服务分层，但整体还不像一个完全统一的 DI 容器驱动系统。

### 建议目标

引入统一的依赖注入组织方式，例如：

- 在 `App.xaml.cs` 中集中注册服务
- 区分：
  - Singleton
  - Scoped（如果后续自己实现模块作用域）
  - Transient
- 统一 ViewModel 和 Service 的创建方式

### 建议结果

做到：

- UI 不手动 new 太多服务
- 模块替换更容易
- 测试时更方便 mock
- 插件和扩展模块更容易接入

---

## 5. A2：统一配置系统

### 当前问题

虽然有 `SystemSettings` 和 `SettingsService`，但后续如果接入真实环境，配置会迅速增多，例如：

- 数据库连接字符串
- 设备地址
- PLC IP
- OPC UA Endpoint
- MQTT Broker
- MES API 地址
- 日志级别
- 插件目录
- 用户权限策略

### 建议目标

建立统一配置管理机制，例如按层拆分：

- `appsettings.json` 风格配置文件
- 用户可修改配置文件
- 设备配置文件
- 配方文件
- 环境配置（开发 / 现场 / 测试）

### 建议后续成果

- 所有外部依赖都不再写死在代码中
- 系统支持不同现场快速切换配置
- 更利于部署和版本管理

---

## 6. A3：明确“模拟实现”和“真实实现”的边界

### 当前问题

现在很多模块是示例化实现，比如：

- 模拟 MES 数据
- 模拟运动轴行为
- 模拟视觉图像
- 模拟设备状态

这对于演示很好，但如果继续迭代，很容易出现：

- 演示代码与正式代码混在一起
- 后续替换成本高
- 真实接入时改动面太大

### 建议目标

建立清晰边界，例如：

- `Simulation` 命名空间 / 目录
- `Real` 命名空间 / 目录
- 统一接口
- 运行时切换模式（演示模式 / 真实模式）

### 示例思路

- `IMesClient`
  - `SimulatedHttpMesClient`
  - `RealHttpMesClient`
- `IDevice`
  - `SimulatedPlcDevice`
  - `RealPlcDevice`
- `IVisionEngine`
  - `SimulatedVisionEngine`
  - `CameraVisionEngine`

这样未来替换更平滑。

---

## 7. A4：补齐编码、注释、命名规范

### 已发现问题

部分文件出现中文乱码注释，说明编码规范可能不一致。

### 建议目标

统一：

- 文件编码为 UTF-8
- 中文注释编码正常
- 命名规范统一
- 公共接口与模型注释补齐
- 文档版本号与代码版本号同步

### 预期收益

- 降低维护成本
- 提升阅读体验
- 方便团队协作
- 便于后续生成文档和自动化分析

---

## 8. A5：完善日志、异常和诊断框架

### 当前已有基础

项目已有日志服务与全局异常保护。

### 下一步建议

增强为：

- 日志分级（Info / Warning / Error / Debug）
- 日志分类（UI / Motion / Vision / MES / Database / Security）
- 文件落盘
- 日志轮转
- 故障上下文记录
- 操作审计日志
- 一键导出诊断包

### 对工业现场的价值

这是后续现场排障和运维的基础。

---

# 阶段 B：把演示模块升级为真实可接入模块

## 9. B 阶段目标

把当前最重要的演示模块逐步替换成“可以接真实业务”的模块。

建议优先级：

1. 数据库
2. MES 通信
3. 设备通信
4. 运动控制
5. 视觉模块
6. 追溯与配方

---

## 10. B1：数据库模块升级

### 当前基础

数据库模块已经支持：

- EF Core
- SQLite
- MySQL
- SQL Server

### 下一步建议

补齐以下能力：

- 数据迁移管理
- 初始化脚本版本控制
- 仓储层边界清晰化
- 历史数据分页查询
- 长周期归档策略
- 大数据量查询优化
- 数据备份与恢复

### 实际价值

这样数据库模块才能从“演示统计”升级为“真实生产记录底座”。

---

## 11. B2：MES 通信升级

### 当前基础

已有：

- HTTP
- MQTT
- OPC UA 客户端结构

### 下一步建议

重点做：

- 定义正式的 MES 接口 DTO
- 增加重试机制
- 增加超时控制
- 增加消息签名或认证机制
- 断线重连
- 失败缓存与补发
- 工单状态同步
- 结果上报幂等控制

### 进一步可做

- 接入真实 MES 沙箱环境
- 支持接口版本适配
- 增加接口调用跟踪页

### 实际意义

这是项目能否真正接入企业系统的关键一步。

---

## 12. B3：设备通信升级

### 当前基础

已有：

- 串口通信服务
- TCP 通信服务
- PLC / 传感器设备对象

### 下一步建议

按照工业接入方式扩展：

- Modbus TCP
- Modbus RTU
- OPC UA 实时订阅
- PLC 厂商协议适配层
- 心跳监测
- 通信质量监控
- 自动重连
- 设备离线告警

### 最终目标

形成统一设备接入抽象层：

- 设备定义
- 点位映射
- 通信驱动
- 状态管理
- 事件通知

---

## 13. B4：运动控制模块升级

### 当前基础

当前运动控制更偏“状态演示与模拟执行”。

### 下一步建议

把模块拆成更清晰的层：

1. 运动抽象接口层
2. 仿真实现层
3. 实际驱动适配层
4. 上位机控制流程层

### 建议新增能力

- 真实控制卡适配接口
- 软限位 / 硬限位状态
- 原点状态与回零策略
- 插补队列
- 程序执行缓冲
- 伺服报警映射
- 急停状态链路
- 手动 / 自动模式切换

### 如果要面向现场

至少要支持：

- 真正的轴使能状态
- 报警复位机制
- 轴状态与 IO 联动
- 更稳定的执行线程模型

---

## 14. B5：视觉模块升级

### 当前基础

当前视觉模块是纯 C# 算法示例。

### 下一步建议

分两步做：

#### 第一步：接真实相机

- 摄像头采集接口抽象
- 实时取图
- 图像缓存管理
- 采图异常处理
- 曝光 / 增益参数配置

#### 第二步：升级检测链路

- ROI 配置
- 模板匹配
- 边缘定位
- 尺寸测量
- 缺陷分类
- 多工位图像任务管理

### 可选方向

- 接入 OpenCV
- 对接工业相机 SDK
- 引入 AI 缺陷分类模型

### 实际意义

这样视觉模块才能从“示例”走向“可用于工站检测原型”。

---

## 15. B6：追溯与配方升级

### 追溯建议

- 增加条码 / SN 扫码接入
- 增加工序模板
- 增加操作人记录
- 增加工艺参数快照
- 增加异常原因分类
- 增加返修记录与复判链路

### 配方建议

- 配方版本管理
- 配方审批状态
- 配方导入导出
- 配方变更日志
- 参数合法性校验
- 产品切换联动加载

### 意义

追溯和配方是“真正制造系统”的核心资产，值得重点强化。

# 阶段 D：平台化、产品化、可复制化

## 22. D 阶段目标

当系统经过前几个阶段后，就可以从“单项目”向“平台产品”发展。

---

## 23. D1：插件化真正落地

### 当前基础

已有 `PluginLoader`。

### 下一步建议

把插件机制真正工程化：

- 明确插件目录规范
- 定义插件元数据
- 插件配置页
- 插件生命周期管理
- 插件加载失败隔离
- 插件版本兼容策略

### 最终目标

让新功能可以通过插件而不是改主程序来增加。

---

## 24. D2：模块市场化 / 产品包化

### 方向设想

后续可把系统拆成多个产品包：

- 标准监控版
- 运动控制版
- 视觉检测版
- MES 对接版
- 全功能工程版

### 好处

- 适应不同客户
- 更利于报价和交付
- 更利于功能组合

---

## 25. D3：多项目模板化

### 建议目标

把当前项目沉淀为一个“工业上位机基础模板”，新项目启动时可以直接复用：

- UI 壳
- 用户权限
- 日志系统
- 配置系统
- 报警系统
- 数据接入框架
- 设备接入框架
- 测试框架

### 价值

这会显著提高以后类似项目的开发效率。

---

## 26. D4：数据平台联动

如果未来走得更远，可以继续扩展：

- 对接 Web 管理后台

# 27. 推荐开发优先级

如果你希望“少走弯路”，建议优先做下面这条顺序：

## 第一优先级：先打底座

1. 统一 DI 与模块注册
2. 统一配置系统
3. 区分模拟实现和真实实现
4. 修复编码/注释规范问题
5. 完善日志与异常体系

## 第二优先级：让最关键模块可接真实业务

6. 数据库模块升级
7. MES 通信升级
8. 设备通信升级
9. 运动控制驱动抽象升级
10. 视觉采图与算法链路升级

## 第三优先级：提升工程可落地性

11. 安装部署体系
12. 健康检查与运行诊断
13. 权限和审计强化
14. 集成测试 / 稳定性测试

## 第四优先级：平台化

15. 插件化落地
16. 模块化产品包
17. 模板化复制能力
18. 向数据平台延伸

# 29. 适合你当前最先做的三件事

如果你现在就准备继续开发，我建议最先做这三件：

### 第一件：整理“模拟实现 vs 真实实现”边界
这是后续所有真实接入工作的前提。

### 第二件：统一配置与依赖注入
这是把项目从“能跑”升级为“可维护”的关键。

### 第三件：优先把 MES / 数据库 / 设备通信三块做实
因为这三块最容易决定系统是否真正具备业务接入价值。

---

# 30. 最终结论

`SmartMES` 当前已经有很好的架构基础，下一阶段不应只是一味加新页面，而应该按下面的顺序发展：

1. **先把底座做稳**
2. **再把关键模块做真实**
3. **再补工程化与运维能力**
4. **最后推进平台化和产品化**

如果按这条路线推进，它完全可以从“优秀的工业演示项目”成长为“真正可落地的智能制造上位机平台”。
