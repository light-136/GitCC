# 1. 先把根本概念讲清楚：什么是上位机，什么是下位机？

## 1.1 上位机是什么

上位机通常是运行在：
- PC
- 工控机
- 工业平板
- 服务器边缘节点
上的软件系统。

它主要负责：
- 人机交互
- 状态展示
- 参数配置
- 流程执行
- 数据采集
- 报警与日志
- 与 MES / 云端 / 数据平台通信
- 协调设备与业务逻辑

## 1.2 下位机是什么

下位机通常是更接近现场设备的一层控制实体，例如：
- PLC
- 单片机
- 运动控制卡
- 工业控制器
- 伺服驱动器
- 视觉控制器
- 温控器
- 采集卡
- 边缘控制器

它主要负责：
- 实时采集
- 实时控制
- IO 逻辑
- 设备联锁
- 硬件动作执行

## 1.3 上位机和下位机的本质分工

你可以这样理解：

### 下位机更偏“实时控制大脑”
负责：
- 快速响应
- 底层时序
- 现场硬件动作
- 信号采集与执行

### 上位机更偏“管理、交互、协调、记录中枢”
负责：
- 人看什么
- 人怎么操作
- 参数怎么管理
- 流程怎么编排
- 数据怎么沉淀
- 如何接上层系统

一句话：

> 下位机更像现场动作层，上位机更像现场管理与交互层。

# 2. 一个真正完整的上位机，应该放在什么体系里理解？

如果从完整工业体系看，一个成熟上位机并不是孤立存在的，它一般位于以下链路中：

```text
ERP / APS / PLM / 云平台
        ↑
      MES / SCADA / 数据平台
        ↑
      上位机 / 工站软件 / 边缘执行端
        ↑
 PLC / 运动控制器 / 工业相机 / IO / 仪表 / 驱动器
        ↑
 传感器 / 电机 / 气缸 / 阀 / 光源 / 执行机构 / 工艺对象
```

所以学习上位机时，你不能只盯着 UI，也不能只盯着 PLC，而要看到：

- 它上接管理系统
- 它下接控制系统
- 它中间做人机桥梁、数据桥梁、流程桥梁

# 9. 上位机与下位机交互系统

这是你这次特别点出来的重点，我单独讲。

## 9.1 上位机与下位机的交互本质

上位机与下位机交互，不只是“发命令”和“收数据”，而是：

- 读取状态
- 写入命令
- 同步模式
- 确认反馈
- 读取报警
- 读取参数
- 写入配方
- 读取执行结果
- 同步节拍与工艺步骤

## 9.2 常见交互方式

### 轮询式交互
上位机周期读取下位机状态。

### 事件式交互
下位机状态变化后主动通知上位机。

### 命令-应答式交互
上位机发命令，下位机回结果。

### 混合式交互
大多数真实系统是这三者混合。

## 9.3 常见交互内容

### 上位机 -> 下位机
- 启动
- 停止
- 急停复位
- 模式切换
- 配方下载
- 参数写入
- 目标位置下发
- 工单信息下发

### 下位机 -> 上位机
- 当前状态
- 报警状态
- 当前位置
- 当前步骤
- IO 状态
- 温度/压力/流量等测量值
- 任务完成信号
- 故障码

## 9.4 为什么交互设计很关键

因为现场很多问题都来自交互设计不清：
- 命令发出去了，但不知道设备是否真的执行
- 状态上报慢，UI 显示与设备不一致
- 下位机报警了，但上位机没及时同步
- 参数已经写入，但操作员不知道是否生效

所以一个成熟上位机必须把“交互协议 + 状态同步 + 可视化反馈”当成一套整体设计。

# 11. 设备抽象与对象建模系统

上位机要管理的不是“报文”，而是“对象”。

## 11.1 为什么要建模

- 相机当前是否可用

而不是：
- 第 12 个字节是不是 0x01

## 11.2 常见对象模型

- 设备模型
- 轴模型
- IO 点位模型
- 相机模型
- 工位模型
- 工单模型
- 产品模型
- 报警模型
- 日志模型
- 追溯模型

## 11.3 建模的意义

建模的本质是：

> 把复杂现实世界变成软件中可管理、可显示、可记录、可扩展的对象体系。

# 12. 状态机与运行模式系统

这是工业软件和普通管理系统最本质的区别之一。

## 12.1 为什么一定要有状态机

因为设备和工艺不是任意可执行的，它们有状态约束。

## 12.2 典型状态

- Idle
- Starting
- Running
- Paused
- Stopping
- Alarm
- Error
- Resetting
- EmergencyStopped
- Homing

## 12.3 运行模式

- 手动模式
- 自动模式
- 调试模式
- 维护模式
- 教学模式
- 离线模拟模式

## 12.4 为什么模式和状态都要分开

状态回答：系统现在在什么运行态。  
模式回答：系统整体按照什么策略运行。

例如：
- 自动模式下的 Running
- 手动模式下的 Idle

- 回零
- 复位
- 点动
- 定位移动

## 13.3 运动控制学：你至少要知道什么

如果你要学运动控制上位机，有几个最基本概念要知道：

### 位置
设备当前在哪。

### 速度
设备移动有多快。

### 加速度
设备速度变化有多快。

### 轨迹
设备不是瞬移，而是沿着路径运动。

### 插补
多轴协调运动时，路径如何连续平滑。

### 回零
让轴回到参考零点，建立坐标基准。

### 限位
防止超出安全范围。

### 使能
伺服/驱动器是否允许动作。

### 跟随误差
命令位置和实际位置之间的偏差。

### 运动模式
- 点位模式
- 连续运动模式
- 插补模式
- 跟随模式

## 13.4 上位机在运动控制中的角色

上位机通常不负责毫秒级闭环控制，但负责：
- 设定目标
- 显示状态
- 管理参数
- 执行程序
- 与工艺流程联动
- 做安全限制和人机交互

# 23. 半导体领域：SECS/GEM（你写成了 SEMC GEM，这里我按行业标准术语科普）

你提到的“SEMC GEM”在半导体领域通常对应的是：

- `SECS`（SEMI Equipment Communications Standard）
- `GEM`（Generic Equipment Model）

更完整常见说法是：
- `SECS/GEM`

这是半导体设备和工厂主机系统之间非常重要的一类标准通信体系。

## 23.1 它是什么

SECS/GEM 是半导体制造设备与工厂主机之间的标准化通信框架，用于实现：
- 设备状态管理
- 控制命令
- 事件通知
- 报警上报
- 变量采集
- 远程命令
- 配方管理
- Lot/批次流程协同

## 23.2 为什么半导体设备特别需要它

因为半导体设备：
- 设备昂贵
- 工艺复杂
- 自动化程度高
- 设备管理标准化要求极强
- 厂务和主机系统要统一接入设备

所以不能靠每家设备厂商自定义接口乱来。

## 23.3 SECS/GEM 上位机视角下最重要的能力

### 设备状态模型
设备处于什么状态，主机必须知道。

### 事件上报
设备发生了什么事件，必须主动通知主机。

### 报警上报
设备报警必须标准化输出。

### 变量采集
主机可读取设备变量、参数、状态值。

### 远程命令
主机可对设备发命令。

### 配方管理
主机与设备之间交换工艺配方。

## 23.4 为什么它值得你学

因为它代表一种“标准化设备-主机通信哲学”，而不仅仅是协议细节。

你会从中学到：
- 如何把设备状态标准化
- 如何把报警和事件统一化
- 如何把设备纳入工厂级自动化体系
