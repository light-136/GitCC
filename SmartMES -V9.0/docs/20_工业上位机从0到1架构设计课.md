# 工业上位机从 0 到 1 架构设计课

> 文档定位：教学型 / 架构方法论型文档  
> 目标：不依赖具体项目，把“如果从零开始设计一个工业上位机，应该怎么想、怎么拆、怎么落地”系统讲清楚。  
> 适用对象：准备自己设计上位机、做毕业设计、做项目原型、做工业软件架构入门的人。

---

# 1. 为什么要学“从 0 到 1 设计”

看懂一个项目，和自己从零设计一个项目，是两种不同能力。

- 前者是理解能力
- 后者是建模能力

如果你只会“看别人写好的代码”，那你学到的是局部实现。  
如果你会“从需求出发，拆成架构和模块”，你才开始真正具备工程设计能力。

所以这份文档的重点不是讲某个类怎么写，而是讲：

> 当你面对一个新的工业场景时，应该如何从 0 开始，把它设计成一个上位机系统。

---

# 2. 先记住一个结论：上位机设计不是从界面开始的

很多初学者一开始会这么做：

1. 想几个页面
2. 画几个按钮
3. 做几个卡片
4. 开始写 UI

这其实是最常见的错误路径。

真正正确的顺序应该是：

1. 先定义业务对象
2. 再定义设备与系统边界
3. 再定义功能域
4. 再定义数据流和状态流
5. 再定义安全与权限
6. 最后再设计界面

一句话：

> 工业上位机应该从“系统结构”开始，而不是从“页面样子”开始。

---

# 3. 从 0 到 1 的第一步：先问场景，不要先问技术

每次设计一个上位机，第一步不是问：
- 用 WPF 还是 WinForms？
- 用什么数据库？
- 用什么图表库？

第一步应该问：

## 3.1 你控制的到底是什么？

- 单台设备？
- 多工位设备？
- 一条产线？
- 实验设备？
- 检测设备？
- 搬运设备？
- 组装设备？

## 3.2 谁在用它？

- 操作员
- 调试工程师
- 设备工程师
- 管理员
- 质量工程师
- 实施人员

## 3.3 现场最关键的问题是什么？

- 要稳定控制？
- 要做质量判定？
- 要追溯？
- 要接 MES？
- 要配置多产品参数？
- 要支持换型？

## 3.4 风险点是什么？

- 撞机风险？
- 参数误改风险？
- 越权操作风险？
- 通信丢失风险？
- 结果错判风险？

你只有把这些问题答清楚，后面的设计才会合理。

---

# 4. 第二步：画系统边界，而不是直接写模块

一个成熟设计者，脑子里一定先有“边界图”。

## 4.1 上位机的典型边界

### 向下边界
接现场设备：
- PLC
- 传感器
- IO
- 运动控制卡
- 相机
- 仪器仪表

### 向上边界
接管理系统：
- MES
- 数据平台
- 报表平台
- 云端服务

### 向内边界
系统内部通常包含：
- UI
- 控制
- 通信
- 数据
- 日志
- 安全
- 流程
- 追溯

## 4.2 为什么边界这么重要？

因为很多混乱项目，根本原因就是边界没想清楚：

- UI 直接操作通信
- 业务逻辑直接写在页面里
- 数据库访问到处都是
- 报警逻辑散落在每个按钮里

边界一乱，项目就会越来越难维护。

---

# 5. 第三步：先画“功能域”，不要急着画页面

在我的知识体系里，一个完整上位机通常先拆成功能域，而不是直接拆成页面。

## 5.1 标准功能域模板

建议最少考虑这 10 类：

1. UI 与导航
2. 通信
3. 设备抽象
4. 控制执行
5. 状态机
6. 调度
7. 报警与日志
8. 配方与参数
9. 追溯与数据
10. 流程编排

可选增强：

11. 视觉检测
12. MES 协同
13. 插件扩展
14. 原生扩展
15. 运维诊断

## 5.2 为什么先拆功能域？

因为页面只是这些能力的展示载体。

比如“报警页面”不是一个功能域，它只是“报警模块”的一个显示界面。

这一步你要学会：

> 先把系统能力拆清楚，再决定这些能力怎么呈现在界面上。

---

# 6. 第四步：设计数据流

工业系统里，最怕的是“功能有了，但数据乱流”。

## 6.1 什么叫数据流？

就是信息从哪里来，经过谁处理，最终存到哪里、显示到哪里。

## 6.2 一个典型上位机的数据流例子

### 状态采集流
设备 -> 通信模块 -> 设备模型 -> ViewModel -> UI

### 结果记录流
检测模块 -> 结果模型 -> 数据库 -> 报表 / 追溯 / MES 上报

### 报警流
安全条件失败 -> 报警模块 -> 日志模块 -> UI 页面 -> 历史存档

## 6.3 为什么数据流必须设计？

如果不设计，你就会遇到：
- 多处重复存数据
- UI 直接查数据库
- 通信层直接更新页面
- 逻辑状态不同步

这类问题在工业系统里会非常致命。

---

# 7. 第五步：设计状态流

这是工业软件和普通管理软件最大的不同之一。

## 7.1 什么叫状态流？

就是系统里的对象从一个状态变到另一个状态的路径。

比如一个设备：

`Idle -> Running -> Paused -> Running -> Alarm -> Resetting -> Idle`

## 7.2 为什么状态流要单独设计？

因为工业软件很少是“点一下就完”，它一定会经历：
- 启动前检查
- 启动后运行
- 运行中暂停
- 报警中断
- 复位恢复

如果状态流没想清楚，后面会出现很多 bug：
- 不能重入
- 状态乱跳
- UI 和设备状态不同步
- 错误无法恢复

## 7.3 状态流设计时要问什么？

- 系统有哪些状态？
- 每个状态允许什么操作？
- 哪些状态之间能切换？
- 切换由什么触发？
- 切换前有哪些守卫条件？
- 切换后谁要收到通知？

---

# 8. 第六步：先做风险设计，再做功能设计

这是工业软件最容易被忽视、却最重要的一步。

## 8.1 为什么风险设计优先？

因为工业项目里最怕的不是“功能少一点”，而是：
- 误操作
- 危险动作无保护
- 状态错误执行
- 参数错误导致产品报废
- 故障后无记录

## 8.2 风险设计要考虑哪些方面？

### 1）权限风险
谁能做什么？

### 2）设备风险
动作是否安全？是否互锁？

### 3）数据风险
关键参数是否会被误改？

### 4）流程风险
中断后是否能恢复？

### 5）诊断风险
出了问题是否能查？

## 8.3 一个成熟系统至少要有

- 急停机制
- 安全互锁
- 页面权限
- 日志留痕
- 异常捕获
- 故障定位辅助

---

# 9. 第七步：再设计架构分层

当你把场景、边界、功能域、数据流、状态流和风险点都想清楚后，才进入架构层。

## 9.1 我建议的标准分层

### 表现层 UI
负责：界面展示、命令绑定、导航。

### 服务层 Services
负责：日志、报警、通信协调、自动化流程、配置等公共服务。

### 业务层 Modules
负责：数据库、MES、运动、视觉、文件处理等业务能力实现。

### 核心层 Core
负责：状态机、调度器、配方、追溯、安全、接口、模型等基础能力。

### 原生层 Native
负责：C++ 运算或厂商 SDK 适配。

### 测试层 Tests
负责：核心行为验证。

## 9.2 为什么这样分层合理？

因为它匹配工业项目的真实复杂度：
- UI 很多
- 公共能力很多
- 业务模块差异大
- 核心机制要复用
- 部分能力需要 native

---

# 10. 第八步：定义核心模块清单

在架构层确定后，你要开始列“必备模块清单”。

## 10.1 最小可用上位机模块

一个最小可用工业上位机，至少应该有：

1. 主界面与导航模块
2. 通信模块
3. 设备状态模块
4. 控制执行模块
5. 报警模块
6. 日志模块
7. 参数配置模块
8. 用户登录与权限模块

## 10.2 进阶工业型模块

如果你要做得更像真实工业系统，还要加：

9. 状态机模块
10. 调度模块
11. 配方模块
12. 追溯模块
13. 流程编排模块
14. 数据统计模块
15. MES 协同模块

## 10.3 高级平台型模块

16. 插件模块
17. 原生扩展模块
18. 健康诊断模块
19. 安装部署模块
20. 审计与权限细分模块

---

# 11. 第九步：再开始设计页面

到这里，才轮到 UI 页面设计。

## 11.1 为什么页面设计放这么后？

因为页面本质上是前面那些设计结果的“壳”。

## 11.2 页面设计建议顺序

### 第一类：总览页
- 首页 / 仪表盘

### 第二类：运行页
- 设备状态页
- 通信页
- 报警页
- 日志页

### 第三类：控制页
- 运动控制页
- 程序执行页
- 手动/自动切换页

### 第四类：管理页
- 用户权限页
- 参数配置页
- 配方管理页
- 数据查询页

### 第五类：专项页
- 视觉检测页
- MES 交互页
- 追溯查询页

## 11.3 页面设计原则

- 关键状态优先可见
- 高风险操作显著隔离
- 信息层级明确
- 避免把过多逻辑塞进一个页面
- 操作结果应有反馈

---

# 12. 第十步：决定哪些逻辑一定不能写进 UI

这是非常重要的经验。

## 12.1 不应直接写进 UI 的内容

- 通信协议细节
- 数据库访问逻辑
- 状态机规则
- 安全互锁判断
- 复杂流程控制
- 追溯记录逻辑
- 设备驱动细节

## 12.2 为什么？

因为 UI 的职责是：
- 显示
- 接收用户操作
- 调用命令
- 接收状态更新

一旦把核心逻辑写进 UI：
- 难测
- 难复用
- 难维护
- 易混乱

---

# 13. 第十一步：设计测试策略

一个上位机从 0 到 1，如果没有测试设计，后期很容易靠手点回归，成本极高。

## 13.1 最值得优先测试的模块

- 状态机
- 调度器
- 安全互锁
- 配方
- 追溯
- 权限逻辑
- 控制核心逻辑

## 13.2 为什么这些优先？

因为这些模块：
- 最关键
- 最容易回归出错
- 最适合做纯逻辑自动化测试

## 13.3 UI 测试要不要做？

要，但通常不是最先做。先把核心逻辑测稳，再逐步加 UI 自动化。

---

# 14. 第十二步：预留演进路线，而不是一次做完一切

很多人设计项目时想一步到位，把所有未来能力都堆进去，结果项目反而难以落地。

更合理的方式是：

## 第一期：做最小可用版本
- 登录
- 首页
- 通信
- 状态显示
- 基本控制
- 报警
- 日志

## 第二期：做工业增强版本
- 状态机
- 调度器
- 安全互锁
- 配方
- 追溯
- 数据统计

## 第三期：做协同增强版本
- MES 接口
- 视觉检测
- 多工位流程
- 数据报表

## 第四期：做平台化版本
- 插件化
- 原生扩展
- 健康诊断
- 部署与运维

这就是从 0 到 1，再从 1 到 10 的思路。

# 15. 一个实际案例：如果让你从 0 设计一个“视觉检测上位机”，你怎么做？

我给你一个具体思路。

## 15.1 先明确场景

目标：做一台视觉检测设备的上位机。

### 用户
- 操作员
- 调试工程师
- 管理员

### 设备
- PLC
- 相机
- 光源控制器
- NG 剔除执行器

### 上层系统
- MES（可选）
- 本地数据库

## 15.2 功能域拆解

必须有：
- UI
- 通信
- 设备状态
- 图像采集
- 检测逻辑
- 结果显示
- 报警日志
- 参数与配方
- 追溯

## 15.3 分层设计

- UI：页面与交互
- Services：日志、报警、配置
- Modules：视觉、通信、数据库
- Core：状态机、配方、追溯、安全

## 15.4 页面设计

- 首页
- 运行页
- 图像页
- 参数页
- 配方页
- 日志页
- 报警页
- 用户页

## 15.5 风险设计

- 相机离线报警
- 检测参数误改限制
- 运行中禁止随意切换关键参数
- NG 剔除失败报警
- 图像与结果必须留档

你看，真正设计一个系统，不是先画页面，而是先把这些逻辑走一遍。

---

# 16. 如果让你从 0 设计一个“运动控制上位机”，你怎么做？

再举一个更偏控制的例子。

## 16.1 场景

目标：做一台三轴运动设备上位机。

### 核心对象
- X/Y/Z 三轴
- 原点传感器
- 限位信号
- 急停按钮
- PLC 或运动控制卡

## 16.2 功能域

必须有：
- 状态展示
- 点动/回零/移动
- 模式切换
- 急停互锁
- 报警日志
- 参数配置
- 手动/自动流程

## 16.3 特别要重视的地方

- 状态机
- 安全互锁
- 线程模型
- 命令幂等性
- 报警恢复逻辑

## 16.4 设计重点

比起页面，你应该先定义：
- `AxisState`
- 允许的状态迁移
- 每个命令在哪些状态下可执行
- Stop/Error/Reset 的边界

这就是工业软件设计思维。

# 17. 从 0 到 1 最容易犯的 10 个错误

## 错误 1：从页面开始，不从系统开始
## 错误 2：没有画边界，导致层次混乱
## 错误 3：UI 直接操作数据库和通信
## 错误 4：没有状态机，全靠 if/else
## 错误 5：没有统一日志和报警体系
## 错误 6：没有急停和互锁优先级设计
## 错误 7：没有权限边界
## 错误 8：没有追溯和数据留痕意识
## 错误 9：没有测试策略
## 错误 10：没有演进路线，想一步到位

如果你能避开这 10 个错误，你的设计能力会提升非常快。

你以后可以直接照着走。

## Step 1：明确场景和对象

这个模板非常重要，你以后可以复用在任何上位机设计里。

# 19. 从 0 到 1 的本质，不是“会写代码”，而是“会建模”

这是全文最重要的一句话之一。

很多人把软件开发理解为“把需求写出来”，但在工业领域更重要的是：

> 把现实世界里的设备、流程、风险、角色、数据和状态，转换成一套清晰的软件模型。

这就叫建模能力。

你能建模：
- 设备才不会乱
- 状态才不会乱
- 流程才不会乱
- 扩展才不会乱

---

# 20. 一句话总结这门“设计课”

> 从 0 到 1 设计工业上位机，真正的起点不是页面、不是控件、也不是数据库，而是先搞清楚：系统控制什么对象、服务什么场景、面临什么风险、需要哪些功能域、如何组织状态与数据，再据此设计分层架构和模块边界；当你掌握了这套顺序，你就真正开始具备工业上位机架构设计能力了。
