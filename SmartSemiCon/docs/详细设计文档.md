### 1.2 粘包/拆包处理

TCP是流式协议，存在粘包（多个包合并到达）和拆包（一个包分成多段到达）问题。

**解决方案：长度前缀协议**

```
数据包格式:
┌──────────────┬────────────────────────┐
│ 4字节(大端序) │      数据负载           │
│   数据长度    │  (length字节)          │
└──────────────┴────────────────────────┘
```

PacketBuilder.Build(data):
```csharp
// 将原始数据封装为带4字节长度前缀的完整包
var packet = new byte[4 + data.Length];
// 写入大端序长度
packet[0] = (byte)(data.Length >> 24);
packet[1] = (byte)(data.Length >> 16);
packet[2] = (byte)(data.Length >> 8);
packet[3] = (byte)(data.Length);
// 复制数据
data.CopyTo(packet, 4);
```

PacketParser.Append(data):
```csharp
// 将收到的字节追加到内部缓冲区
// 循环尝试提取完整包：
//   1. 缓冲区长度 >= 4 → 读取长度字段
//   2. 缓冲区长度 >= 4 + length → 提取一个完整包
//   3. 继续尝试（处理粘包）
//   4. 缓冲区不足 → 等待下次数据到达（处理拆包）
```

### 1.4 命令-响应匹配

```csharp
// 使用 ConcurrentDictionary<int, TaskCompletionSource<byte[]>> 实现
// 发送请求时：
int msgId = Interlocked.Increment(ref _messageIdCounter);
var tcs = new TaskCompletionSource<byte[]>();
_pendingRequests[msgId] = tcs;
await SendAsync(packet);
var response = await tcs.Task.WaitAsync(timeout);

// 收到响应时：
if (_pendingRequests.TryRemove(msgId, out var tcs))
    tcs.SetResult(responseData);
```

### 2.1 协议栈

```
┌─────────────────────────────────────────┐
│  GEM合规层 (GEM Compliance)              │
│  - 控制状态机 (Online/Offline)           │

│  - 报警管理 (S5F1)                       │
├─────────────────────────────────────────┤
│  SECS-II消息层 (Message Layer)           │
│  - SecsItem 编解码                       │
│  - Stream/Function 路由                  │
│  - 消息处理器注册                         │
├─────────────────────────────────────────┤
│  HSMS-SS传输层 (Transport Layer)         │
│  - 10字节消息头                          │
│  - Select/Deselect 握手                  │
│  - Linktest 心跳                         │
│  - 事务管理 (SystemBytes)                │
├─────────────────────────────────────────┤
│  TCP Socket                              │
└─────────────────────────────────────────┘
```

### 2.2 HSMS消息头格式（10字节）

```
字节 0-3:  消息长度（大端序，含消息头后续6字节+消息体）
字节 4-5:  SessionID / DeviceID
字节 6:    HeaderByte2 (W-bit + Stream)
字节 7:    HeaderByte3 (Function)
字节 8:    PType (0=SECS-II)
字节 9:    SType (0=Data, 1=Select.req, 2=Select.rsp, ...)
字节 10-13: SystemBytes（事务ID，用于匹配请求和响应）
```

## 5. 视觉-运动协同

### 5.1 坐标转换系统

```
像素坐标(px, py)  ──▶  仿射变换  ──▶  机械坐标(wx, wy)

仿射变换矩阵 (3×3):
┌           ┐   ┌    ┐   ┌    ┐
│ a  b  tx  │   │ px │   │ wx │
│ c  d  ty  │ × │ py │ = │ wy │
│ 0  0  1   │   │ 1  │   │ 1  │
└           ┘   └    ┘   └    ┘

wx = a*px + b*py + tx
wy = c*px + d*py + ty
```

### 5.2 9点标定流程

```
标定网格 (3×3):

  P1 ── P2 ── P3
  │      │      │
  P4 ── P5 ── P6
  │      │      │
  P7 ── P8 ── P9

流程:
1. 移动到P1 → 采集图像 → 检测Mark点 → 记录(像素, 机械)坐标对
2. 移动到P2 → 采集图像 → 检测Mark点 → 记录坐标对
3. ... 重复9次
4. 最小二乘法计算仿射变换矩阵
5. 计算标定RMS误差
```

### 5.3 对位补偿流程

```
AlignAsync 流程:

1. 采集图像 ──▶ CaptureAsync()
2. Mark点检测 ──▶ FindMarkAsync()
3. 坐标转换 ──▶ PixelToWorld(px, py)
4. 计算偏差 ──▶ dx = target_wx - detected_wx
                dy = target_wy - detected_wy
5. 运动补偿 ──▶ MoveRelativeAsync(dx, dy)
```

### 6.1 任务调度器架构

```
┌─────────────────────────────────────────────┐
│  IndustrialTaskScheduler                     │
├─────────────────────────────────────────────┤
│  ScheduledTask:                              │
```

### 6.2 优先级定义

| High | IO信号处理 | 独立线程 | 10-50ms |
| Normal | 状态刷新、数据采集 | 线程池 | 100-500ms |
| Background | UI刷新、日志 | 线程池 | 500-5000ms |

### 7.2 状态转换规则

每个转换规则包含：
- **Guard条件**：是否允许转换（如：必须不在报警状态才能初始化）
- **转换动作**：转换时执行的异步操作
- **事件发布**：通过事件总线通知其他模块

```csharp
// 规则示例：Idle → Init
new StateTransitionRule
{
    Trigger = "Initialize",
    From = DeviceState.Idle,
    To = DeviceState.Init,
    Guard = () => _alarmService.ActiveAlarms.Count == 0,
    Action = async () => { /* 初始化逻辑 */ }
}
```

### 7.3 紧急停止处理

EmergencyStop 是特殊状态，只能通过 Reset 恢复：
1. 立即停止所有运动轴
2. 关闭所有输出
3. 发布急停事件
4. 状态机进入 EmergencyStop 状态
5. 只有手动确认 Reset 才能恢复

## 8. 报警管理详细设计

### 8.1 报警代码分段

```
报警代码范围          所属模块           典型场景
────────────────────────────────────────────────
1000 ~ 1999          运动控制           跟随误差/软限位/伺服报警
2000 ~ 2999          视觉系统           相机连接超时/算法失败
3000 ~ 3999          通讯模块           TCP断连/PLC超时/SECS超时
4000 ~ 4999          安全系统           急停信号/安全门开/光幕触发
5000 ~ 5999          流程控制           配方参数异常/生产超时
```

### 8.2 报警去重机制

```csharp
// ConcurrentDictionary 按报警代码去重
// 同一报警代码在活跃期间不会重复触发
if (!_activeAlarms.TryAdd(alarm.Code, alarm))
{
    // 报警已存在，忽略重复触发
    return;
}
// 新报警，写入历史 + 发布事件
_alarmHistory.Add(alarm);
_eventBus.Publish(new AlarmTriggeredEvent { Alarm = alarm });
```

### 8.3 报警等级与系统联动

```
Fatal → DeviceStateMachine.ForceState(EmergencyStop)
         → AxisManager.EmergencyStopAllAsync()
         → SecsGemService.SetAlarmAsync()

Heavy → DeviceStateMachine.FireAsync("Alarm")
         → SecsGemService.SetAlarmAsync()

Light → 暂停当前流程，等待自动恢复

Warning/Info → 仅记录日志，不影响生产
```

## 10. 用户认证详细设计

### 10.1 密码安全

```csharp
// SHA256 哈希计算
private static string HashPassword(string password)
{
    var bytes = Encoding.UTF8.GetBytes(password);
    var hash = SHA256.HashData(bytes);
    return Convert.ToHexString(hash).ToLowerInvariant();
}

// 登录验证
public async Task<bool> LoginAsync(string username, string password)
{
    var user = _users.Values.FirstOrDefault(u => u.Username == username);
    if (user == null || !user.IsEnabled) return false;
    if (user.PasswordHash != HashPassword(password)) return false;

    CurrentUser = user;
    user.LastLoginTime = DateTime.Now;
    _eventBus.Publish(new UserLoggedInEvent { User = user });
    return true;
}
```

### 10.2 权限层级

```
Administrator (管理员) > Engineer (工程师) > Operator (操作员)

权限检查：当前用户角色 >= 要求角色
HasPermission(UserRole.Engineer)
  → CurrentUser.Role >= UserRole.Engineer
  → Admin: true, Engineer: true, Operator: false
```
