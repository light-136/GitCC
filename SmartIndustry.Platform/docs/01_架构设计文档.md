# SmartIndustry.Platform 架构设计文档

> 版本：1.0 | 作者：架构师 | 日期：2026-05-25

---

## 一、项目定位

**SmartIndustry.Platform** 是一个基于 C# / .NET 9.0 / WPF 的多人协同工业智能平台。

**目标**：以一个完整的工程实例，展示从架构师视角如何设计一个涵盖"文件、网络、数据库、界面、算法、运动控制、视觉"全领域的工业软件系统。

**核心学习价值**：
- Clean Architecture（洁净架构）的分层实践
- C# 语言深度特性（泛型、异步、事件、表达式树、值对象、record）
- WPF + MVVM 的完整工程化实践
- 多人协同的实时状态同步设计
- 工业级硬件抽象（运动控制 + 机器视觉）

## 四、分层职责详解

### 4.1 Domain 层（领域核心）

| 组件 | 职责 | C# 特性体现 |
|------|------|-------------|
| Entities | 业务实体，含领域行为 | 抽象类继承、软删除、乐观并发 |
| ValueObjects | 不可变值对象 | record类型、运算符重载、Equals/GetHashCode |
| Events | 领域事件定义 | record继承、不可变性 |
| Interfaces | 全系统接口契约 | 泛型接口、异步模式、事件委托 |
| Enums | 业务枚举 | 强类型枚举 |
| Specifications | 规约查询模式 | 表达式树、Lambda、组合模式 |

### 4.2 Infrastructure 层（基础设施）

| 模块 | 功能 | 关键技术 |
|------|------|----------|
| Database | EF Core + SQLite ORM | DbContext、泛型仓储、审计拦截、查询过滤 |
| FileSystem | 文件读写/压缩/监控 | async/await、FileSystemWatcher、ZIP |
| Network/Tcp | TCP客户端/服务端 | Socket异步、自动重连、心跳、粘包拆包 |
| Network/Http | HTTP REST客户端 | HttpClient、JSON序列化 |
| Network/Mqtt | MQTT消息队列 | MQTTnet、发布订阅 |
| Logging | 双通道日志 | Serilog、ConcurrentQueue环形缓冲 |
| EventBus | 进程内事件总线 | 泛型、ConcurrentDictionary、弱引用 |
| Configuration | JSON配置热重载 | FileSystemWatcher、泛型反序列化 |

### 4.3 Hardware 层（硬件抽象）

| 模块 | 功能 | 关键技术 |
|------|------|----------|
| Motion/Drivers | 运动控制卡驱动 | 接口抽象（可替换品牌）、模拟卡仿真 |
| Motion/AxisController | 单轴控制封装 | 状态机、软限位、异步到位等待 |
| Motion/AxisManager | 30轴统一管理 | 全局急停、轴组管理 |
| Motion/Scheduler | 运动任务调度 | BlockingCollection、4优先级、生产者-消费者 |
| Motion/Interpolation | 直线/圆弧插补 | 数学算法、微段分割 |
| Motion/Profiles | 梯形/S曲线规划 | 运动学公式、七段S曲线 |
| Vision/Engines | 视觉引擎 | 接口抽象（可替换Halcon/OpenCV） |
| Vision/Pipeline | 视觉处理管线 | 责任链模式、步骤编排 |
| Vision/Calibration | 九点标定 | 最小二乘法、仿射变换矩阵 |
| Vision/Algorithms | 图像处理算法 | 纯数学实现（Sobel、大津法等） |
| IO | 数字/模拟IO | 边沿检测、防抖、周期轮询 |
| Coordination | 视觉-运动协同 | 坐标转换、对位补偿、飞拍模式 |

### 4.4 Application 层（应用编排）

| 模块 | 功能 | 关键技术 |
|------|------|----------|
| StateMachine | 设备状态机 | 规则表驱动、守卫条件、状态转移回调 |
| AlarmManager | 报警管理 | 5级报警、事件驱动、历史查询 |
| RecipeManager | 配方管理 | 版本链、JSON持久化、激活/切换 |
| UserManager | 用户权限 | SHA256哈希、角色分级、权限校验 |
| TaskScheduler | 工业任务调度 | 4优先级、CancellationToken生命周期 |
| SecsGem | SECS/GEM通讯 | HSMS传输、SECS-II编解码、GEM状态机 |
| Collaboration | 多人协同 | TCP广播、状态同步、会话管理 |

### 4.5 UI 层（WPF 表现层）

| 模块 | 功能 | 关键技术 |
|------|------|----------|
| Views | 12+ 功能页面(XAML) | DataTemplate、样式、触发器 |
| ViewModels | MVVM数据绑定 | CommunityToolkit.Mvvm、ObservableProperty |
| Navigation | 页面导航 | DI容器驱动、ContentControl绑定 |
| Converters | 值转换器 | IValueConverter |
| Resources | 主题/多语言/图标 | ResourceDictionary、DynamicResource |
| Controls | 自定义控件 | UserControl、DependencyProperty |

## 五、核心设计模式骨架

### 5.1 事件驱动架构（EDA）

```
  模块A                    EventBus                    模块B
    │                         │                          │
    │  Publish(AxisMoved)     │                          │
    ├────────────────────────►│                          │
    │                         │  Notify(AxisMoved)       │
    │                         ├─────────────────────────►│
    │                         │                          │
    │                         │  Notify(AxisMoved)       │
    │                         ├─────────────────────────►│ 模块C
    │                         │                          │
```

**所有模块通过事件总线解耦**，不直接相互引用。

### 5.2 MVVM 数据流

```
  View (XAML)          ViewModel (C#)           Service/Model
      │                     │                        │
      │ {Binding Prop}      │                        │
      │◄───────────────────►│                        │
      │                     │                        │
      │ Command执行         │  调用Service           │
      ├────────────────────►├───────────────────────►│
      │                     │                        │
      │                     │  返回结果              │
      │ PropertyChanged     │◄───────────────────────┤
      │◄────────────────────┤                        │
      │ UI自动更新          │                        │
```

### 5.3 硬件抽象策略（可替换）

```
                   IMotionCard（接口，定义在Domain）

          ┌─────────────┼──────────────┐
          ▼             ▼              ▼
```

## 九、关键设计决策

### 决策1：为什么选 Clean Architecture 而不是传统三层

| 对比 | 传统三层 | Clean Architecture |
|------|----------|-------------------|
| 依赖方向 | UI → BLL → DAL | 全部指向Domain核心 |
| 数据库 | DAL是基础 | 数据库是细节，可替换 |
| 测试性 | 依赖数据库才能测 | Domain纯逻辑，独立测 |
| 硬件替换 | 耦合具体实现 | 接口抽象，DI切换 |

### 决策2：为什么用事件总线而不是直接调用

模块间直接调用会形成网状依赖。事件总线使模块只依赖事件定义（在Domain中），不依赖彼此。运动完成后，视觉模块、UI模块、日志模块都能独立响应，互不影响。

### 决策3：为什么硬件层用模拟实现

工业软件开发不能等硬件到位。模拟实现允许：
- 无硬件环境下完整开发和测试
- 替换真实硬件只需实现接口 + DI切换
- 仿真数据支持UI全功能演示

## 十三、风险分析

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 模拟实现与真实硬件行为差异 | 仿真通过但真机失败 | 接口契约严格定义行为规范，模拟卡模拟真实时序 |
| 事件总线性能瓶颈（高频事件） | UI卡顿 | 事件合并（throttle）、异步处理、UI线程 Dispatcher |
| 多人协同状态冲突 | 数据不一致 | 乐观锁 + 最后写入胜出(LWW) + 冲突检测通知 |
| EF Core SQLite 并发写入限制 | 数据丢失 | WAL模式 + 写入队列串行化 |
